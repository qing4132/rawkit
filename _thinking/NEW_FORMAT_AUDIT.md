# 「在 RAW 与 JPEG 之间发明新中间格式 + 新编辑器 + 新 UX」—— 冷启动审视

> 假装我是 Discord 群里被 ping 进来的资深 VC + indie dev 老兵。
> 这是我第一次听说这个 pitch,作者背景、之前讨论我都不知道。
> 我只看这个 idea 本身。

---

## 0. 一句话第一反应

> "技术上漂亮、智识上吸引人、商业上接近自杀;作为爱好项目'凑合做'尚可,作为养活路线**不投**。"

下面拆开为什么。

---

## 1. 这个 pitch neutral 重述

某人提议:**发明一种新的静态影像中间文件格式**,介于 RAW 和 JPEG 之间。
- 类比视频界 15 年前发明的 log 编码 + intermediate codec (ProRes/DNx)
- 锁死一批默认参数("约定大于配置"),牺牲一部分灵活性换取小文件、快速编辑、清晰工作流
- 同步发明一个使用此格式的**新型编辑器**,且**编辑 UX 也要"前所未见"**
- 由一位 solo dev 独立设计、实现、推广

**所以这个 pitch 实际上是三个独立的大赌注叠在一起**:

1. 发明一个新文件格式(格式赌)
2. 围绕它做一个新编辑器(产品赌)
3. 编辑 UX 也要前所未见(UX 赌)

任何一个单拎出来都是 3 年级别的工程 + 商业 + 设计挑战。三个叠在一起,solo dev 做完之前精力耗尽的概率超过 80%——这是历史经验,不是我个人悲观。

---

## 2. 真正打动我的部分(诚实赞美)

**a. 视频 log 这个类比选得准**。video 界确实是被 4K RAW 的代价压垮才发明 log/intermediate codec,这套组合释放了整个行业的工作流。提出这个类比的人至少在两个领域都有真正的技术理解,不是 random "I'll make a photo editor" 的小白。

**b. "90% 的人在 90% 的时候要 90% 的功能"这个 framing 是真正好的产品哲学**。绝大多数失败的工具都是因为想满足 100% 而臃肿成屎。如果只看哲学这一项,我会想多了解作者。

**c. "约定大于配置 + 锁死默认"是反 LrC/Photomator 臃肿的真武器**。Lightroom 滑块多到没人全用过;Photomator 想轻又什么都做;darktable 配置无底洞。**真的有空间立一个opinionated 极简编辑器**。

**d. 自用驱动 + 顺手开源(暗示)的姿态健康**。比"我要做下一个 Lightroom 颠覆者"那种 pitch 健康十倍。这种姿态做出过 SQLite (D. Richard Hipp 个人项目)、curl (Daniel Stenberg 个人项目)。

**这些都是真东西。我不是来打压热情的,但下面要说的也是真东西。**

---

## 3. 一听就警铃的部分

### 警铃 A: 新文件格式是软件史上最强的"创始人坟场"信号之一

过去 20 年由 solo dev 或小团队发明并获得真实采用的新媒体文件格式,列表近乎空白。**真实成功的格式都来自巨型主体**:

| 格式 | 出身 |
|---|---|
| WebP | Google |
| AVIF | AOMedia 联盟(Google/Cisco/Intel/Mozilla/Netflix...) |
| HEIF | MPEG |
| JPEG XL | JPEG 委员会,5+ 年才得到部分浏览器支持 |
| ProRes | Apple |
| DNG | Adobe(20 年后相机厂支持率仍 < 30%) |
| ProRAW | Apple |
| Fujifilm 压缩 RAF | Fuji 自己 |

**模式很清晰**: 文件格式采用率是**相机厂或平台厂的决策**,不是软件开发者的决策。一个 solo dev 发明的格式,**结构上**没有被任何下游接受的路径——除非那个 solo dev 同时控制相机硬件、操作系统、或主流编辑软件之一。

→ 这意味着这个格式天然**只能是自用**。这本身可以,但**应该被诚实地承认**,而不是含糊地"先自用再说"——后者经常是失败的合理化前置。

### 警铃 B: 视频 log 类比的成立条件,在照片域几乎一条都不成立

|  | 视频 log 成立时(2008–2015) | 静态摄影 2026 |
|---|---|---|
| 痛点强度 | 4K RAW 在当时电脑上**完全无法回放/编辑** | R5 RAW 在 M 系列 Mac 上**秒开、滑块实时** |
| 推动方 | 相机厂(Sony S-Log, Panasonic V-Log, Canon Log)| 无相机厂 |
| 工作流方 | 后期厂(Resolve, Avid)需要更小代理 | 无后期厂主动需要 |
| 行业意愿 | 电影/广电行业有强烈标准化动力 | 摄影行业历史上抗拒新格式 |
| 经济压力 | 4K 工作站每天烧机房电费 | 个人 SSD 几百块 1TB |

**当年驱动视频 log 的所有结构性压力,在今天的摄影领域几乎都不存在**。这意味着:即使你做出了完美的中间格式,**没有外部力量替你推它**——你要自己一个人推它。

### 警铃 C: 这个 pitch 的真正内核很可能不是"新格式",而是"Smart Preview as a first-class working file"

Lightroom 2012 起就有 Smart Previews:1MP DNG,用作慢电脑/出门时的"轻量代理"。Capture One、ON1 都有类似机制。

**这个 pitch 中性翻译,就是: "把 Smart Preview 从 fallback 升级成主编辑目标"**。

这个 reframe 之后,你会发现:
- 这不是一个新媒体格式,是一个**工作流约定** + 已有压缩格式的使用习惯
- 现有工具加几行代码就能模仿大部分价值主张
- "新格式"这层包装**增加了工程成本但没有相应增加用户感知价值**

→ 这是 VC 必问的"这真的需要新格式吗" 灵魂拷问。如果答案是"不一定需要,但我想做",那就该承认这是审美/智识驱动,而非产品需要。

### 警铃 D: "90/90/90" 隐藏了最致命的问题

**所有依赖"烘焙某些决定"的格式,都死在那 10% 的烘焙错误用户身上**。

- 烘焙白平衡 → 极端混合光照场景救不回来 → 用户抓狂
- 烘焙降噪 → 高 ISO 细节永久丢失 → 用户抓狂
- 烘焙镜头矫正 → 老镜头/特殊镜头 profile 错 → 永久变形
- 锁死分辨率 → 用户某天想出大画幅打印 → "我只有这个低分辨率的中间格式" → 用户离开
- 锁死位深 → 用户某天需要救严重过曝/欠曝 → 救不回 → 用户离开

每一个 10% 失败场景,**用户记仇 10 年**。摄影师对工具的怨念是结构性的——他们记得 1998 年某次坏掉的 print。

补救方案"那种时候你再回 RAW 重转一次"看起来合理,但:
- 这意味着用户必须**永远保留 RAW**——你的"小文件"价值主张被对半砍
- 用户被训练成"重要照片不信任你的格式"——你的格式被自动降级成 proxy
- 反复重转 = 反复重做编辑决定 = 双倍工作量

**这条不是格式细节问题,是哲学问题。一个不能容错的格式,在容错本能极强的人群里推不动。**

### 警铃 E: 三个赌注叠加 = 失败概率乘法

格式赌 60% 失败 × 编辑器赌 70% 失败 × UX 赌 70% 失败 = **整体失败率 ~92%**。

这不是悲观。Solo dev + 三个大赌注同时打,通常的归宿:
- 6 个月内做出格式 CLI 转换器(技术好玩,作者投入)
- 9 个月卡在编辑器 GUI(GUI 是真活,作者大概率没做过)
- 12 个月精力耗尽,项目成"另一个半成品 repo"
- 18 个月之后偶尔回来 commit,3 年后 archive

我见过这个模式 50 次以上。**它和作者智商无关,和"三个赌注同时打"的结构有关**。

### 警铃 F: "我要做前所未见的东西"这句话本身是个信号

资深 indie dev 听到这句通常的反应:**warm caution**。

真正成功的"前所未见"通常是**事后被命名为前所未见**(SQLite/Notion/Figma 当年都不觉得自己在做颠覆,只是在解一个具体问题)。**事前就追求"前所未见"的项目,通常被这个野心本身压垮**——因为它把"是否新颖"当成产品的核心评价标准,而用户根本不在乎新颖,只在乎好用。

不是说追求新颖不对,是说**把"新颖"放在第一位会让你拒绝所有现成解,即使现成解最适合**。这个心态做研究可以,做产品危险。

## 4. VC 视角真正会问的问题

如果作者来 pitch 我做投资,我会要回答以下问题再说话:

1. **用户是谁?** 不要回答"像我这样的人"。要回答"具体多少人、在哪能找到、他们目前用什么、为什么会换"。
2. **他们的当前痛点是什么?** 真痛到能让他们换工具?Lightroom 用户的"痛"是 Adobe 订阅(已被 Photomator 等接走),不是文件格式。
3. **付费意愿在哪?** 摄影爱好者付费率 < 5%。专业付费用户要的是 RAW 全保真,不是"中间格式"。**这个产品打的是哪一群?**
4. **现有玩家(Apple ProRAW、Smart Preview、HEIF)解决了同问题的多少?** 如果是 70%,你要回答"我那 30% 改进值多少"。
5. **格式的 chicken/egg 怎么破?** 没人支持 → 没用户;没用户 → 没人支持。除非"自己永远是唯一用户",承认这个就 OK,但要明说。
6. **3 年时间线现实吗?** Solo dev,有别的收入吗?家庭支撑?6–12 月没收入能撑吗?

**这些问题如果回答不出具体的、不挥手的答案,VC 都会 pass**。这不是恶意,是模式识别——这种回答不上来的 pitch 失败率 > 90%。

---

## 5. Indie dev 朋友视角(温度上来一点)

VC 帽子摘掉,作为做过几个 indie 项目的人对作者说的话:

**这个项目作为"养活自己"的赌注**:
- 12 个月养活概率 < 10%
- 24 个月养活概率 < 20%
- 不要假设这个能养活你

**作为"个人智识探索 + 工程练习 + 顺手开源"的项目**:
- 完全 OK,甚至很棒
- 你会学到 RAW pipeline / 色彩科学 / 文件格式设计 / GUI 工程——这些**单拎出来都是稀缺技能**,能反哺你之后的咨询/接活/求职
- 如果你享受这个过程本身,做的价值已经在过程里给付了

**如果你坚持做,我会强烈建议**:
1. **明确分离"格式实验"和"编辑器产品"**。先做格式 + CLI 转换器 + 一个 viewer(2–3 个月,基本是你舒适区)。**不要 day 1 就承诺做 GUI 编辑器**。
2. **设硬 deadline**: 12 周交出 v0(CLI 完整可用),不交就退出。indie 项目最大的杀手是无 deadline。
3. **承认这是自用项目**,从一开始就别幻想"做出来再说"。"做出来再说"的项目在 indie 圈死亡率 95%。
4. **金钱基线另外解决**(接活、咨询、写技术内容)。这个项目脖子上不挂养活的绳子,反而才能做好。
5. **不要追求"前所未见"**。追求"我自己用得爽 + 代码干净 + 6 个月后还想打开"。新颖会作为副产品出现,不要主动追。

## 6. 一个 VC 会下注的版本(假如必须)

如果硬逼我从这个 pitch 改造出一个**可投资**的版本,我会这样改:

**砍掉**:
- 砍掉"发明新格式"(转为工作流约定 + 现有格式如 JPEG XL/DNG 子集)
- 砍掉"前所未见 UX"(用既有滑块/直方图,只在交互节奏上极简)
- 砍掉"全栈编辑器"(只做 LrC 旁挂工具,补它最弱的一段)

**留下**:
- 留下"约定大于配置"的设计哲学
- 留下"opinionated 极简工作流"的定位
- 留下"local-first"的信仰
- 留下作者的 dogfood 能力

**改造结果**: 一个轻量、opinionated、本地的 LrC companion 或 Photomator 替代,使用标准格式但**工作流极简到反 LrC**。这个我可能会投——它是上面 pitch 的 1/5 工程量,但保留了 80% 的灵魂,且有清晰用户路径。

**但作者明确说了"我要前所未见的东西",所以他不会接受这个改造**。这是 VC 视角 vs 创作视角的根本张力。

---

## 7. 我个人最终的话

如果我是这位作者的朋友,而不是 VC,我会说:

> "你这个想法在智识上漂亮、在工程上是好挑战、在审美上对你 dogfood,**这些都真**。
> 但它**几乎不可能养活你**,这一点也真。
> 不要把它和养活捆在一起。**它属于'我此生要做的几个东西之一',不属于'我的下一个 SaaS'**。
> 如果你接受这个定位——花 1 年做出 v0 自己用爽 + 顺手公开,养活靠别的渠道——那就动手,不要再讨论了,讨论会杀掉它。
> 如果你需要它必须养活你才肯做,那就**别做**,去做上一轮那两个候选(catalog search / RAW 前置处理器)里你不太喜欢但能赚钱的那个。
> 你不能两个都要。"

---

## 8. 一句送给自己(VC 帽子)

这个 pitch 是这类 pitch 的典型样本:**作者智识高、品味好、工程力强、idea 漂亮——所有'我会喜欢的特征'都满足。但作为产品,它命中了我所有'我不投'的 pattern**(新格式、新编辑器、新 UX、solo、无清晰付费用户、3 年时间线、技术驱动而非用户驱动)。

VC 的纪律就是: **我喜欢什么 ≠ 该投什么**。

我对作者的尊重是真诚的。我说不投也是真诚的。两件事可以同时为真。
